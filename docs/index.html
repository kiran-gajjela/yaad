summary

Your WhatsApp export, turned into something you can actually ask questions of

Point yaad at any exported chat and it builds a private, queryable archive on your machine — ask what was decided, who said what, who's most active, or what happened last week, and it answers from the real messages, cited. Chat data breaks naive RAG two ways: single messages are useless retrieval units, and half the interesting questions are aggregations no embedding can answer. So yaad classifies every question first — search, analytics, or both — then runs hybrid retrieval or guarded SQL accordingly. One SQLite file per chat, one local model, no cloud account, no vector database to manage.

case study

Six friends, 1,800km, one WhatsApp group

Everything below is real model output against a real-shaped export — no cherry-picked transcripts, no edited answers.

verified live

Five capabilities, five real questions

Each card below is one actual exchange with the Ladakh group's export, run against the local gemma4:e4b model. Nothing here is paraphrased for effect.

semantic retrieval

Finds meaning, not just keywords

"what happened with that permit problem back in april?"

Correctly narrated the full Inner Line Permit crisis — the new sponsor-agent rule, the panic, Vikram sourcing a local operator, the ₹500/person resolution — despite the chat never using the word "problem" anywhere.

BM25 ⊕ dense embeddings, fused with Reciprocal Rank Fusion

multi-hop reasoning

Connects two separate facts

"why did karan drop out, and who replaced him?"

"Karan dropped out because he tore his ACL playing football... Farah replaced Karan's slot for the trip." Two unrelated messages, six weeks apart, correctly linked into one answer.

Single retrieval pass, not an agentic multi-step loop

context-grounded

Reports disagreement, doesn't resolve it for you

"where are they staying the last night in Leh?"

Correctly presented all three live positions — homestay, hotel, decide-later — because the group genuinely never settled it. No invented resolution.

System prompt requires per-claim (sender, date) citation

structured analytics

Exact numbers, not estimates

"who is the most active person in this group?"

"Rahul is the most active person with 24 messages" — a real COUNT(*) GROUP BY sender query the model wrote itself against a documented schema, not a guess from reading excerpts.

Read-only connection, row-capped, one retry on error

temporal reasoning

"Last week" means the real last week

"tldr, summarize last week"

Resolved the date range from the system clock, not model guesswork, then pulled every matching message chronologically — a Sanjay return, a vendor confirmation, a poll — instead of a similarity-ranked sample that could miss half of it.

Deterministic date parsing; exhaustive retrieval for bounded ranges

architecture

Export in, cited answer out

Every stage below runs on-device by default. Nothing leaves the machine unless you explicitly opt into a cloud model.

WhatsApp export (.txt)
Exported by hand from the phone — no API, no account linking, no scraping.
Parserlocal
Android + iOS formats, DD/MM auto-inference, system-message detection kept out of the sender field.
Sessionizer
New session after a 45-minute gap — separates unrelated bursts of chat.
Chunker
25-message sliding window, 5-message overlap — sized in messages, not tokens.
indexed two ways, in one file
SQLite FTS5
BM25 lexical search — names and places are strong anchors in chat.
Dense vectors
all-MiniLM-L6-v2, 384-dim, brute-force cosine — instant at personal-chat scale.

no separate vector database — both indexes live in the same .db file

Routerlocal LLM
Classifies intent — search / analytics / both — with a keyword-heuristic fallback if the LLM call fails.
question intent splits the path
Hybrid retrieval
BM25 ⊕ dense, fused via RRF. Bounded date ranges pull everything, not a ranked sample.
Guarded text-to-SQL
One read-only statement, row-capped, one auto-retry — against a documented schema only.
Synthesisgemma4:e4bcloud optional
Grounds every claim, cites (sender, date), and says "not mentioned" instead of inventing.
Answer, on your screen
Cited and private by construction — nothing left the machine to produce it.
